Procedural Knowledge Ontology: Description back to ToC

The main class of the ontology is Procedure, which is defined as a sequence of actions to be executed in order to achieve a desired outcome. Such outcome can be expressed in terms of a Procedure Type (e.g., a maintenance activity), that is the type of procedure being executed on a Target (e.g., a production line for a specific type of machines). The class Procedure is declared as a subclass of both Plan, and Resource (intended as any entity that can be included in a catalog.) If someone wants to manage version control, and keep track of updates and revisions to the same procedure, it is possible to link an "abstract" procedure to its versioned procedures (hasVersion), which are in a sequential order (nextVersion, previousVersion). A Procedure can be marked as a template (isTemplate) and can be linked to procedures derived from it (hasTemplate, isTemplateOf), so that reusable procedure blueprints can be managed alongside executable procedures. A Procedure can be associated with a ProcedureStatus (e.g., draft, validated, approved). A Procedure is linked to all its Steps with the property hasStep. A Step groups one or more Actions (from a human - requiresAction) or Functions (from an algorithm/software - requiresFunction) to execute a portion of a Procedure, possibly with the use of some Tools (requiresTool). A Step can be decomposed as a set of multiple steps itself (Multistep). The class StepVerification represents the way in which the execution of a Step can be verified. When different agents (e.g., a junior vs a senior technician) perform the same procedure, we may need to specify the ExpertiseLevel of each (Multi)Steps, meaning that the step is targeted at an agent with that level of expertise. This allows for different formulations and levels of detail of the same steps. A certain Procedure can be executed one or multiple times, by one or more agents, at a certain time. This is represented by the classes of ProcedureExecution and StepExecution, which are defined as subclasses of Activity, and thanks to which we can track the execution of the procedure as a whole, and as a composition of executions of the individual steps. The ProcedureExecution can be associated with a ProcedureExecutionStatus (e.g., in progress, completed, paused). A Change Of Status captures status updates of any relevant entity (fromStatus, toStatus) as timestamped activities. The UserFeedbackOccurrence represents the case in which an Agent leaves a feedback either on the procedure or the procedure execution. The UserQuestionOccurrence allows to keep track of the questions that an Agent may come up with while performing the procedure, and to the possible Resource that allowed to address the question. The IssueOccurrence models the occurrence of an Error that may be encountered by an Agent at some point during execution, and allows to record the cause, as identified by the Agent, and the solution that has been applied by the Agent. Such Error may be associated with an errorCode, and may be addressed by a specific fallback (Multi)Step included in the Procedure. An Agent, while interacting with a procedure, can play a certain Role (e.g., editor, supervisor, user). The class RoleInTime makes it possible to model a role that an agent may have and to restrict itto a particular PeriodOfTime. A Procedure can reference (property references) different Resources, like documents and images that constitute relevant documentation; it can be extracted from (wasExtractedFrom) a Resource, as in the case of e.g. a PDF containing some text describing a procedure to execute. A Requirement can be attached to a Procedure, to a Step, or to a Transition between steps, and can be typed through a Requirement Type. During execution, requirement fulfillment can be represented through Requirement Satisfaction instances, linked either to a ProcedureExecution or to a StepExecution, and associated with a Requirement Satisfaction Level (e.g., not satisfied, partially satisfied, satisfied). A Transition explicitly captures the passage from one step to another (fromStep, toStep), and can be linked to the Explanation about the possible reasons why the transition may happen. Instead, at the execution level, the Explanation represents the actual reason why the transition happened. The ontology also supports governance and access control through policies and evaluation requests. A Procedure (Type) can be linked to one or more Policies (hasPolicy), and policy rules can specify the assignee Role (assigneeRole). EvaluationRequest represents the requested action (evaluatedAction) and target (evaluatedTarget), i.e. a Procedure (Type), and is linked to an EvaluationRequestStatus (e.g., pending, accepted, rejected) and an optional requestStatusMotivation to document the reason behind request status.
